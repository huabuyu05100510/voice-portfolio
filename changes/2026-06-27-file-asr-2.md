# 录音文件识别 2.0 — File ASR 2.0 (火山引擎 bigmodel 异步)

> 日期: 2026-06-27
> 范围: 客户端 + 后端 + CSS + 测试 + 文档
> 主题: 把 1h 录音 / 会议室录音 / 采访文件接入火山引擎录音文件识别 2.0,
>       与实时转写合并到同一 Hero 区展示.

---

## 1. 业务目标

当前 voice-portfolio 只有实时 ASR (录音 → WebSocket → partial/final 流).
但实际场景里:

- 1h 会议室录音 / 演讲录音 — 太长, 实时流成本太高
- 离线访谈文件 — 用户拿到 mp3/wav 后想批量识别
- 视频文件 (.mp4 / .mov) — 需要先抽音再转写

火山引擎 **录音文件识别 2.0** 提供异步任务模型, 适合上述场景:
- 提交 → 排队 → 识别 → 拉取结果 (支持轮询 + 回调)
- 单文件 ≤ 100MB, 时长 ≤ 12h
- 多说话人 / 词级时间戳 / 句子分段 等价于实时 v3 协议

我们要做的是: 在主区加 "文件识别 2.0" Tab,
上传后展示任务列表, 完成后把结果 merge 进 transcriptionReducer,
让用户能在同一个 Hero 区看到文件识别 + 实时转写的结果.

## 2. 技术方案

### 2.1 协议选型

参考项目内已有的 `volcengine_session.py` / `volcengine_engine.py` 的 v3 实时协议,
录音文件识别 2.0 使用相同的鉴权格式:

```
Authorization: Bearer; {token}   (注意分号 + 空格)
X-Api-Key:    {app_id}
X-Cluster:    {cluster}
```

调用三段式:
1. `POST /api/v3/recognitions/bigmodel` — 提交 (body: `{url, model, ...}`)
2. `GET  /api/v3/recognitions/bigmodel/query?task_id=...` — 查状态
3. `GET  /api/v3/recognitions/bigmodel/result?task_id=...` — 拉结果 (可选, query 也会带)

任务状态枚举 (我们规范化后): `queued / running / done / failed`

### 2.3 客户端状态机 (per task)

```
uploading → submitted → running → done
                    ↓           ↓
                  failed    failed
                    ↓
                cancelled (用户主动取消)
```

轮询策略: `setTimeout(poll, 2000)`, 最多重试 3 次 (指数退避),
失败 / 超时把 task 标 `failed` 并显示重试按钮.

### 2.4 合并到实时转写 (核心点)

文件识别结果 = `Utterance[]` + 词级时间戳, 与实时 v3 协议的 `final` 帧结构一致.
我们把 `parse_result()` 产物装进 `TranscriptionResult`, 通过 dispatch:

```ts
dispatch({
  type: 'TRANSCRIPT_FINAL',
  result: {
    text: fileResult.text,
    isFinal: true,
    isCumulative: false,        // 标记"独立源", reducer 不影响流
    utterances: fileResult.utterances,
    words: [...],
    speakers: [...],
  },
  isCumulative: false,
});
```

reducer 现有的 `TRANSCRIPT_FINAL` 分支直接吃这个 action, 不需要改动.
这就是"零侵入挂载"的关键.

### 2.5 CSS 设计参考

dropzone / task list 视觉对标:

- **Otter.ai**: 大号虚线 dropzone, 居中 icon + 提示文案
- **Dropbox Record**: 紧凑型 list, 状态徽章 + 进度条
- **Apple Translate**: 状态色 = 系统色 (绿/蓝/红/橙)

颜色映射:
- `submitted` → 蓝色 (待开始)
- `running` → 橙色 + 进度条 (识别中)
- `done` → 绿色 (完成)
- `failed` → 红色 (失败)
- `cancelled` → 灰色 (取消)

### 2.6 可观测性

- OTel span: `file_asr.submit` / `file_asr.poll` / `file_asr.merge`
- 结构化日志:
  - `[FileASR] submit {filename, size_bytes, format}`
  - `[FileASR] status {task_id, status, progress}`
  - `[FileASR] merge {local_id, utterances_count}`
- Prometheus metrics (在 `/metrics/summary` 里追加):
  - `file_asr_submit_total`
  - `file_asr_completed_total`
  - `file_asr_failed_total`
  - `file_asr_poll_latency_seconds`

## 4. 端到端测试计划

1. 启动 NestJS 后端 + Vite 前端
2. 准备一个 30s 的 mp3 文件 (用 ffmpeg 生成或语音合成)
3. 在 UI 上传 → 验证:
   - 文件元信息显示正确
   - 任务状态: uploading → submitted → running → done
   - Hero 区出现该文件的 utterances (与实时结果并列展示)
   - 词级时间戳可点击 (跳转到对应音频位置, 后续 feature)
4. 错误路径:
   - 上传 .txt → 立即报错 "格式不支持"
   - 上传 200MB 文件 → "文件过大, 最大 100MB"
   - 上传后手动取消 → 状态变 cancelled, 不再轮询
   - 服务端 500 → 状态 failed, 显示重试按钮 → 点击重试 → 重新轮询

## 6. 风险与限制

1. **objectURL 同源限制**: `URL.createObjectURL` 的 blob 只能在同一 origin 访问,
   Vite proxy 配置已处理 (开发环境).
2. **内存**: 大文件 arrayBuffer 解码可能 OOM, 已加 try/catch.
3. **网络抖动**: 轮询超时 3 次后放弃, 用户可手动重试.
4. **凭据安全**: API key 仅在服务端配置, 前端永远拿不到.
